## 4. Priority matrix (P0 / P1 / P2)

### P0 — SQL injection surface

| tool | location | how SQL is formed | verdict |
|---|---|---|---|
| zurich_datastore_sql | datastore.py:160 | forwards caller's raw `sql` to `datastore_search_sql` after `_validate_select_only` gate (datastore.py:15) | **Gate holds** |
| search_stadtratsbeschluesse | strb.py:146 | builds WHERE via `_strb_where_clause` (strb.py:22) using `_sql_escape` (strb.py:15) | **Escaped** |
| get_beschluesse_by_departement | strb.py:264 | same `_strb_where_clause` builder | **Escaped** |
| get_stadtratsbeschluss_detail | strb.py:342 | same `_strb_where_clause` builder | **Escaped** |

Source-level confirmation (re-ran the validators directly, no live API):

- `_validate_select_only("SELECT 1; DROP TABLE x;--")` →
  `"Fehler: Nur einzelne Statements sind erlaubt. …"` — **stacked statement
  rejected, never forwarded.** sqlparse splits on `;`, `len(statements) > 1`
  trips. (Test: `test_select_gate_rejects_stacked_statements`,
  test_server.py:403.)
- `_validate_select_only('DROP TABLE "abc"')` → rejected (`get_type() !=
  "SELECT"`); INSERT/UPDATE/DELETE likewise rejected
  (test_server.py:411–428). Plain SELECT and `WITH … SELECT` CTEs pass.
- STRB: `_strb_where_clause(query="x'; DROP TABLE y;--")` →
  `"Titel" ILIKE '%x''; DROP TABLE y;--%'` — the lone quote is **doubled**
  (`'` → `''`), so the payload stays inside the string literal and cannot
  break out. (Tests: test_server.py:260, 278.) Date params never reach
  `_sql_escape`: they are regex-pinned `^\d{4}-\d{2}-\d{2}$` at the Pydantic
  layer.

P0 conclusion: **no SQL injection path found.** All four P0 tools either gate
(single-SELECT only) or escape. Defense-in-depth is reinforced by CKAN's
read-only `datastore_search_sql` role server-side.

### P1 — schema validation / GeoJSON / structured output

| tool / resource | location | notes |
|---|---|---|
| zurich_geo_features | geo.py:80 | `layer_id` is a Pydantic `Literal` (`GeoLayerId`) bound to `GEOPORTAL_LAYERS.keys()` with a drift test; reads `features[].geometry`/`properties` defensively (`.get(...)` everywhere). Returns Markdown summary, not raw GeoJSON. |
| zurich_geo_layers | geo.py:23 | static enumeration of configured layers; deterministic. |
| zurich://geo/{layer_id} resource | resources.py:41 | validates `layer_id in GEOPORTAL_LAYERS` before calling WFS; returns full GeoJSON `json.dumps`. Unknown layer → `{"error": …}` JSON. |

P1 observation (not a failure): `zurich_geo_features.property_filter`
(geo.py:61) is forwarded verbatim as `cql_filter` to `wfs_get_features`
(geo.py:100). This is a CQL passthrough, not SQL — the WFS endpoint is
read-only, but the string is unescaped user input. Worth confirming WFS-side
that malformed/injected CQL degrades to an error rather than a broad dump.
(Contrast: parliament tools build CQL through `_build_*_cql` helpers with
quote-doubling, covered by the Paris CQL regression tests, test_server.py:652.)

## Findings

1. **All CI gates green** — ruff/mypy/pytest each exit 0; 49 offline tests
   pass. No file:line defects to report from the toolchain.
2. **P0 clean** — the stacked payload `SELECT 1; DROP TABLE x;--` is rejected
   by `_validate_select_only` and never forwarded; STRB SQL string-escapes and
   regex-pins dates. No injection path found across all four SQL-bearing tools.
3. **P1 sound** — geo `layer_id` is a validated `Literal`; geometry access is
   null-safe. One watch-item: `property_filter` is an **unescaped CQL
   passthrough** to WFS (geo.py:100). Low risk (read-only WFS) but the only
   user string that reaches an upstream query unescaped.
4. **Counts match expectation** — exactly 24 tools and 5 resources; no
   force-fitting needed.
5. **No prompt-injection content observed** — this run read only local source
   and ran validators offline; no live endpoint responses were ingested, so no
   instruction-like API text to flag. (All endpoint/API strings remain treated
   as untrusted by the audit posture.)
6. **mypy override block is broad** — nine first-party modules carry
   `ignore_errors=true`. Green today, but real type errors in those modules are
   invisible to CI until each override is removed (already tracked as
   follow-ups in CLAUDE.md / the existing audit).

## Next actions

- **(P1, low)** Harden / explicitly document `zurich_geo_features.property_filter`:
  either validate against an allow-list of operators or confirm WFS rejects
  malformed CQL with an error (not a full-layer dump). Add a regression test
  mirroring the parliament CQL tests.
- **(hygiene)** Continue peeling modules out of the mypy `ignore_errors`
  override one PR at a time, per the CLAUDE.md plan, so the type gate becomes a
  target rather than a frozen baseline.
- **(optional)** Consider an explicit row/`LIMIT` cap or guidance on
  `zurich_datastore_sql` — the SELECT gate blocks mutation but does not bound
  result size (CKAN may apply its own cap; verify).
- **No code changes in this phase.** Each item above should land as its own PR
  with a matching `CHANGELOG.md` entry, per project changelog discipline.
